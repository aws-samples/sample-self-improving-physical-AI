## Problem Statement

Build a Physical AI proof-of-concept using the XGO2 Lite robot dog to demonstrate AWS edge AI capabilities for an internal workshop. The demo must show: human text command → cloud reasoning (Bedrock) → edge vision (on-device ML) → motor actuation, proving the "perception → decision → action" loop that differentiates Physical AI from virtual-only GenAI demos.

**Target audience:** BD team + manager (decision to greenlight full workshop investment)
**Timeline:** 2-4 weeks to working POC
**Success gate:** Any working Physical AI loop (e.g., "wave right leg" based on vision input), not necessarily full object manipulation

## Business Context

- **Why this matters:** Current AWS workshops demo static ML (classification, chatbots). This would be the first Physical AI demo showcasing edge orchestration, multi-model pipelines, and real-world actuation.
- **Decision makers:** Manager + BD team (multiple stakeholders, no single named decision maker yet)
- **Durability:** Aligns with broader AWS edge AI strategy, survives personnel changes
- **Risk:** Competitor landscape unknown - have not surveyed what other AWS teams or cloud providers are demoing for robotics/Physical AI

## Constraints

**Must showcase these AWS services:**
- AWS IoT Core (device connectivity, MQTT telemetry)
- AWS Greengrass (edge runtime, component deployment)
- Amazon Bedrock (language model for command decomposition)
- Amazon SageMaker (model training/deployment pipeline)
- Edge ML inference (vision model running on Raspberry Pi CM4)

**Hardware:**
- XGO2 Lite robot dog (12 DOF: 4 legs, 1 arm, 1 claw)
- Raspberry Pi CM4 onboard
- 320x240 camera
- 2-inch LCD display
- Serial control via `xgolib.py`

**Non-negotiable:**
- On-device ML inference (vision model runs on Pi, not cloud) - this is the workshop narrative
- 2-4 week timeline
- Must work reliably for live demo

## Functional Requirements

### FR1: Hardware Validation (Phase 0 - CRITICAL PATH)
**MUST validate before building cloud pipeline**

1.1. Serial communication with XGO2 motors via `xgolib.py`
- Verify: Can send commands and receive responses
- Test: Make robot walk forward 10cm, backward 10cm, turn 90°
- Exit criteria: 5/5 successful command executions

1.2. Arm and claw actuation
- Verify: Arm can extend/retract, claw can open/close
- Test: Pick up a lightweight object (foam ball, empty can)
- Exit criteria: Repeatable grasp-and-release

1.3. Camera capture during motor operation
- Verify: Camera feed remains stable (no frame drops, corruption) while motors running
- Test: Capture 30 frames while robot walking
- Exit criteria: 30/30 clean frames, no I/O blocking

1.4. On-device ML inference performance
- Verify: Pi CM4 can run a vision model at usable FPS
- Test: Run MobileNetV2 or YOLOv5n on camera feed
- Exit criteria: ≥5 FPS with motor control loop running concurrently
- Fallback: If <5 FPS, identify bottleneck (model size? resolution? CPU vs. EdgeTPU?)

**STOP CONDITION:** If any Phase 0 validation fails, re-scope before proceeding to cloud pipeline.

### FR2: Command Decomposition (Bedrock Integration)

2.1. Web interface for text command input
- User enters natural language command (e.g., "wave your right leg", "walk forward and stop when you see red")
- Web app sends command to Bedrock API

2.2. Bedrock reasoning layer
- Input: Natural language command
- Output: Structured action sequence in JSON format
- Example: "wave right leg" → `[{"action": "detect_object", "params": {"color": "none"}}, {"action": "leg_motion", "params": {"leg": "right_front", "pattern": "wave", "cycles": 3}}]`
- Model: Claude 3.5 Sonnet via Bedrock (or latest available)

2.3. MQTT publish to IoT Core
- Web app publishes structured command to topic: `xgo2/commands/{device_id}`
- QoS 1 (at least once delivery)
- Telemetry: Log command timestamp, Bedrock latency, MQTT delivery confirmation

### FR3: Edge ML Vision (Greengrass Component)

3.1. Vision model deployment via Greengrass
- Model: Object detection (YOLO, MobileNet-SSD, or SageMaker-trained custom model)
- Deployment: Model artifact in S3 → Greengrass component downloads and loads
- Inference: Runs on Pi, processes camera frames
- Output: Detected objects with labels, bounding boxes, confidence scores

3.2. Vision trigger conditions
- Detect specific objects (e.g., "red ball", "person", "hand gesture")
- Detect environment state (e.g., "obstacle ahead", "clear path")
- Frame rate: Minimum 5 FPS (see FR1.4 validation)

3.3. Vision output to reasoning layer
- Publish detections to local topic (Greengrass local pub/sub)
- Format: `{"timestamp": ..., "detections": [{"label": "red_ball", "bbox": [x, y, w, h], "confidence": 0.92}]}`

### FR4: Action Execution (Motor Control)

4.1. Greengrass component subscribes to commands
- Listens to MQTT topic `xgo2/commands/{device_id}`
- Receives structured action sequence from Bedrock

4.2. Reasoning loop (action planner)
- Reads current vision detections
- Executes next action in sequence
- Example: If action is "wave_leg" and no vision trigger needed → execute immediately
- Example: If action is "approach_red_ball" → wait for vision to detect red ball, then calculate approach vector

4.3. Motor command translation
- Maps high-level actions to `xgolib.py` serial commands
- Example: "wave right leg" → `xgolib.move_leg(leg=1, height=20, cycles=3)`
- Example: "walk forward" → `xgolib.set_velocity(vx=15, vy=0, vyaw=0)`

4.4. Execution feedback
- Publish telemetry to IoT Core: action started, action completed, errors
- Topic: `xgo2/telemetry/{device_id}`
- Payload: `{"action": "wave_leg", "status": "completed", "duration_ms": 1500}`

### FR5: End-to-End Demo Scenario

Minimum viable demo for BD approval:

**Scenario 1: Vision-triggered action**
- Human types: "Wave your right leg when you see a red object"
- Bedrock decomposes: [detect red object, wave leg when detected]
- Robot: Camera scans, detects red ball, waves right front leg 3 times
- Telemetry: Web UI shows detected object + action confirmation

**Scenario 2: Simple action (no vision)**
- Human types: "Walk forward 5 steps"
- Bedrock decomposes: [walk forward, count steps, stop]
- Robot: Walks forward, stops after 5 steps
- Telemetry: Web UI shows step count + completion

**Acceptance:** Either scenario works reliably (3/3 successful runs) in front of BD team.

## Non-Functional Requirements

### NFR1: Reliability
- Demo success rate: ≥80% (4/5 runs work without manual intervention)
- Graceful degradation: If vision model fails, robot should not crash (log error, continue or abort safely)

### NFR2: Latency
- Command → first motor action: <5 seconds
- Vision detection → action trigger: <2 seconds
- Acceptable for demo (not production)

### NFR3: Observability
- Web UI shows: command sent, Bedrock output, vision detections, action status
- CloudWatch logs: All component logs streamed from Greengrass
- IoT Core telemetry: Command latency, action completion, error rates

### NFR4: Reproducibility
- All Greengrass components version-controlled
- Model artifacts in S3 with version tags
- Infrastructure as code (CDK or Terraform) for IoT Core setup
- Deployment: Single script to deploy updated components to robot
