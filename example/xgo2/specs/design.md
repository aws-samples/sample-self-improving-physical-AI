## Problem Statement

Build a Physical AI proof-of-concept using the XGO2 Lite robot dog to demonstrate AWS edge AI capabilities for an internal workshop. The demo must show: human text command → cloud reasoning (Bedrock) → edge vision (on-device ML) → motor actuation, proving the "perception → decision → action" loop that differentiates Physical AI from virtual-only GenAI demos.

**Timeline:** 2-4 weeks to working POC  
**Decision gate:** BD team + manager approval to invest in full workshop development  
**Success metric:** Any working Physical AI loop (vision-triggered action OR simple action)

## Approaches Considered

### Approach A: Cloud-Heavy (Minimize Edge Complexity)

**Summary:** Vision model runs in cloud (Rekognition or Bedrock Vision), only motor control on device. Greengrass streams camera frames, receives high-level commands.

**Effort:** M (2-3 weeks)  
**Risk:** Low  

**Pros:**
- No on-device ML ops complexity
- Powerful cloud models (better accuracy)
- Fast iteration (no edge deploy cycle)

**Cons:**
- **Fails workshop narrative** (doesn't showcase edge ML)
- 500-1000ms latency (frame upload + inference)
- Requires WiFi (no offline operation)

**Reuses:** Existing Greengrass pipeline, camera code

### Approach B: Edge-First with Pre-trained Model (RECOMMENDED)

**Summary:** Deploy pre-trained YOLO or MobileNet-SSD to Pi via Greengrass ML component. Skip SageMaker training for POC, use COCO-trained model. Bedrock for command decomposition, all vision inference local.

**Effort:** M (2-3 weeks)  
**Risk:** Medium (Pi performance unknown)

**Pros:**
- Showcases edge ML (meets workshop requirement)
- No training pipeline overhead for POC
- Proven models (YOLO/MobileNet well-documented)
- 50-200ms inference latency
- Can upgrade to Approach C post-POC if approved

**Cons:**
- Pre-trained model limited to COCO labels (80 classes: person, ball, cup, etc.)
- May not detect workshop-specific objects
- If Pi <5 FPS, harder to debug without training pipeline

**Reuses:** Greengrass ML examples, xgolib motor control, camera code

### Approach C: Full Edge ML Pipeline (SageMaker → Greengrass)

**Summary:** Train custom object detection model on SageMaker (fine-tune YOLOv5), deploy via Greengrass. Full AWS edge ML story: data collection → labeling → training → deployment → inference.

**Effort:** L (3-4 weeks)  
**Risk:** High (Pi performance + training pipeline + data collection all unproven)

**Pros:**
- Complete AWS edge ML showcase
- Custom model for workshop-specific objects
- Best narrative for "how customers do this at scale"
- Reusable workshop pattern

**Cons:**
- Need labeled training data (50-200 images/class)
- SageMaker training time (hours to days)
- Model optimization (quantization, pruning) adds complexity
- High timeline blow-out risk

**Reuses:** Nothing substantial (greenfield ML pipeline)

## Recommended Approach

**Approach B: Edge-First with Pre-trained Model**

**Why:**
- Balances workshop narrative (edge ML) with timeline risk
- Showcases Greengrass ML deployment without training overhead
- COCO dataset covers common demo objects (person, ball, sports equipment)
- Pre-trained YOLO/MobileNet proven to run on Pi CM4
- If BD approves, Approach C becomes the full workshop (attendees train models)

**Architecture:**

**Component breakdown:**

1. **Web UI (Amplify):**
   - Text input field
   - Real-time telemetry display (command sent, Bedrock output, vision detections, action status)
   - Video stream (optional, for demo visibility)

2. **Bedrock integration (Lambda):**
   - Receives text command from web UI
   - Calls Bedrock API (Claude 3.5 Sonnet)
   - Prompt: "Decompose this robot command into structured action sequence. Output JSON."
   - Publishes structured command to IoT Core MQTT

3. **Greengrass component (Python):**
   - **Vision module:** Loads YOLO/MobileNet-SSD, runs inference on camera frames (target: ≥5 FPS)
   - **Reasoning module:** Subscribes to MQTT commands, reads vision detections, plans action sequence
   - **Motor control module:** Translates high-level actions to xgolib serial commands
   - **Telemetry module:** Publishes action status to IoT Core

4. **Model deployment:**
   - Model artifact: Pre-trained YOLOv5n or MobileNet-SSD (COCO dataset)
   - S3 bucket: `s3://{account}-{region}-greengrass/models/yolov5n.pt`
   - Greengrass ML component downloads model on deploy
   - Inference engine: PyTorch or TFLite (depends on model choice)

**Phased implementation:**

**Phase 0: Hardware Validation (Days 1-3)**
- Serial motor control (walk, turn, wave leg)
- Arm/claw actuation
- Camera stability during motor operation
- ML inference performance (target: ≥5 FPS)
- **GATE:** If <5 FPS or motors fail, reassess scope

**Phase 1: Local Execution (Days 4-7)**
- Greengrass component deployed
- Vision model running locally (hardcoded test: detect person, log to console)
- Hardcoded action sequence (no Bedrock yet)
- Telemetry flowing to IoT Core
- **GATE:** Local loop working (vision → action)

**Phase 2: Cloud Integration (Week 2)**
- Web UI deployed (Amplify)
- Bedrock command decomposition
- MQTT flow: Web → IoT Core → Greengrass
- Demo scenario working end-to-end (1 scenario, 3/3 runs)
- **GATE:** Ready for internal demo

**Phase 3: Refinement (Weeks 3-4)**
- Both demo scenarios working (vision-triggered + simple action)
- Web UI polished
- Failure modes handled (graceful degradation)
- Architecture diagram + demo script

**Tech stack:**
- Vision: YOLOv5n (lightweight, Pi-optimized) or MobileNet-SSD v2
- Inference: PyTorch (YOLOv5) or TFLite (MobileNet)
- Greengrass: Python 3.9, IPC pub/sub for local topics
- Web UI: React + Amplify Hosting
- Backend: Lambda + API Gateway (Bedrock calls) + IoT Core MQTT

**Why not Approach A:** Fails workshop narrative (no edge ML showcase)

**Why not Approach C:** Too much risk for POC timeline. SageMaker training pipeline is the right move AFTER BD approval, not before. Approach C becomes the workshop content - attendees collect data, train models, deploy to robots.

## Open Questions

1. **YOLOv5n vs. MobileNet-SSD?** (Recommend: YOLOv5n - better Pi performance benchmarks, active community)
2. **PyTorch vs. TFLite inference?** (Recommend: PyTorch for YOLOv5, defer optimization)
3. **Web UI hosting: Amplify vs. EC2?** (Recommend: Amplify - faster deployment)
4. **Greengrass component: Monolithic or split?** (Recommend: Monolithic for POC - vision+motor in one component)
5. **Who specifically approves this in BD?** (Need named decision maker by end of Week 1)
6. **What are competitors demoing?** (2-hour landscape scan before Week 2)
7. **If Pi can't hit 5 FPS, what's the fallback?** (Options: reduce resolution, switch to MobileNet, use EdgeTPU accelerator)
